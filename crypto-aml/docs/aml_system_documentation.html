System Overview

What is Anti-Money Laundering (AML)?

Anti-Money Laundering (AML) refers to a set of laws, regulations, and procedures designed to prevent criminals from disguising illegally obtained funds as legitimate income. Money laundering typically occurs in three stages:

  • Placement — Introducing illicit funds into the financial system
  • Layering — Obscuring the trail through complex transactions
  • Integration — Reintroducing funds into the legitimate economy

Why AML Matters in Blockchain

Blockchain networks like Ethereum are pseudonymous — transactions are publicly visible but addresses are not directly tied to real-world identities. This creates a unique challenge: the data is transparent, but attribution is difficult. Criminals exploit this by:

  • Using mixers and tumblers to break transaction trails
  • Rapidly moving funds through dozens of wallets (layering)
  • Interacting with DeFi protocols to obscure fund origins
  • Creating circular transaction loops to simulate legitimate activity
Key Insight: Unlike traditional banking AML where institutions hold KYC data, blockchain AML must rely entirely on on-chain behavioral patterns, graph analysis, and heuristic clustering to identify suspicious entities.

System Goals

This system is designed to achieve four primary objectives:

  1. Detect suspicious transactions — Flag high-value transfers, contract interactions, and anomalous patterns
  2. Cluster addresses — Group addresses likely controlled by the same entity using six distinct heuristics
  3. Assign risk scores — Quantify suspicion on a 0–100 scale based on behavioral evidence
  4. Generate alerts and reports — Surface actionable intelligence for AML analysts

Component Descriptions

Data Ingestion Module

The EthereumExtractor connects to an Ethereum RPC endpoint (Alchemy or public node) and fetches raw blocks including all transactions, receipts, and event logs. Each block is stored idempotently in MongoDB's raw_blocks collection, keyed by block_number. A parallel raw_transactions collection stores one document per transaction for fast address-based lookups.

Transaction Processor (ETL Transform)

The Transformer reads raw blocks from MongoDB, normalizes all fields (hex → decimal, timestamps → UTC datetime, addresses → lowercase), computes risk flags (risk_flag_high_value, risk_flag_contract), and writes clean CSVs to data/processed/. These are then loaded into MariaDB for SQL queries and Neo4j for graph analysis.

Clustering Engine

The ClusteringEngine is the core analytical component. It reads transactions via a BlockchainAdapter, builds a directed multigraph using NetworkX, runs six pluggable heuristics to find address links, merges those links using Union-Find, and produces scored ClusterResult objects.

Labeling Engine

Labels are assigned automatically based on which heuristics fired. Each heuristic maps to a semantic label (e.g., loop_detection → possible_layering). Labels are stored alongside clusters in MongoDB and propagated to Neo4j address nodes.

Risk Scoring Module

Each cluster receives a 0–100 risk score computed from the weighted sum of heuristic contributions plus a size bonus. Scores ≥ 70 are High risk, 40–69 are Medium, and below 40 are Low.

Alert Generation System

The FastAPI /api/alerts endpoint surfaces individual flagged transactions with cluster names, risk scores, and reasons. The /api/clusters endpoint exposes the full clustering engine output.

Visualization Layer

The React frontend provides five views: Transaction Table, AML Alerts, Analytics (with clustering summary), Address Clusters, and an interactive Transaction Network graph powered by React Flow / XYFlow.

NetworkX — Graph Processing

NetworkX is used to build an in-memory directed multigraph of Ethereum addresses and transactions. It provides the algorithmic foundation for all six clustering heuristics.

import networkx as nx

# Create a directed multigraph (allows parallel edges)
G = nx.MultiDiGraph()

# Add address nodes
G.add_node("0xabc...", blockchain="ethereum")

# Add transaction edge with metadata
G.add_edge("0xabc...", "0xdef...",
    key="0xtxhash...",
    value_eth=1.5,
    timestamp=1700000000,
    is_contract_call=False)

# Query: find all successors (receivers) of an address
receivers = list(G.successors("0xabc..."))

# Detect cycles (for loop heuristic)
cycles = list(nx.simple_cycles(nx.DiGraph(G)))

Union-Find — Cluster Merging

The Union-Find (Disjoint Set Union) data structure efficiently merges address pairs from multiple heuristics into clusters. It uses path compression and union-by-rank for near-constant time operations.

# O(α(n)) amortised — effectively O(1) per operation
uf = UnionFind(["0xA", "0xB", "0xC", "0xD"])
uf.union("0xA", "0xB")  # A and B are in the same cluster
uf.union("0xB", "0xC")  # Now A, B, C are all in the same cluster
clusters = uf.clusters()
# → {"0xA": {"0xA", "0xB", "0xC"}, "0xD": {"0xD"}}

Core Data Structures

TxRecord — Normalized Transaction

All blockchain data is normalized into a TxRecord dataclass before entering the clustering engine. This abstraction allows the same engine to work with Ethereum today and Bitcoin (UTXO) in the future.

@dataclass
class TxRecord:
    tx_hash: str           # Unique transaction identifier
    block_number: int      # Block height
    timestamp: float       # Unix epoch seconds
    from_address: str      # Sender (empty for coinbase/UTXO)
    to_address: str        # Receiver (empty for OP_RETURN)
    value_eth: float       # Amount transferred in ETH
    is_contract_call: bool # True if calldata present
    input_method_id: str   # First 4 bytes: "0xa9059cbb" = ERC-20 transfer
    gas_used: int          # Gas consumed

ClusterResult — Output Object

Every cluster produced by the engine is represented as a ClusterResult dataclass containing all information needed for display, storage, and reporting.

@dataclass
class ClusterResult:
    cluster_id: str              # "C-A3F2B1..." (SHA1 of sorted addresses)
    addresses: List[str]         # All member addresses
    risk_score: float            # 0.0 – 100.0
    labels: List[str]            # ["possible_mixer", "fan_out"]
    heuristics_fired: List[str]  # ["loop_detection", "contract_interaction"]
    indicators: Dict[str, object]# {"total_eth_volume": 45.2, "size": 7, ...}

BlockchainAdapter — Abstraction Interface

class BlockchainAdapter(ABC):
    """Interface every blockchain type must implement."""

    @property
    @abstractmethod
    def blockchain_type(self) -> str:
        """Return 'account' or 'utxo'."""

    @abstractmethod
    def iter_transactions(self, **kwargs) -> Iterator[TxRecord]:
        """Yield normalised TxRecord objects from the data source."""

Clustering Engine — Deep Dive

The clustering engine is the analytical heart of the system. It takes a stream of normalized transactions, builds a graph, runs six independent heuristics to find address pairs that should be grouped together, merges those pairs using Union-Find, and produces scored cluster objects.

Engine Orchestration Flow

ClusteringEngine.run()
    │
    ├─ 1. adapter.iter_transactions() → [TxRecord, ...]
    │
    ├─ 2. build_graph([TxRecord]) → nx.MultiDiGraph
    │       Nodes = addresses
    │       Edges = transactions (with metadata)
    │
    ├─ 3. For each heuristic:
    │       heuristic.find_links(G) → [(addr_a, addr_b), ...]
    │       uf.union(addr_a, addr_b)
    │       record which heuristic fired for this pair
    │
    ├─ 4. uf.clusters() → {root: {members}}
    │
    ├─ 5. For each cluster with size >= min_cluster_size:
    │       _compute_indicators() → behavioral stats
    │       score_cluster() → ClusterResult
    │
    └─ 6. Sort by risk_score descending
          Optional: persist to MongoDB + Neo4j

① Behavioral Similarity behavioral_similarity Score: +10

Concept: Two addresses that repeatedly interact with the same set of counterparties are likely controlled by the same entity. For example, two wallets that both regularly send to the same exchange deposit addresses, DeFi contracts, or peer wallets are behaviorally similar.

Algorithm:

  1. Build a counterparty set for each address: union of all addresses it sent to AND received from
  2. For every pair of addresses (A, B), compute the intersection of their counterparty sets
  3. If |intersection| ≥ N (configurable, default 3), link A and B
def find_links(self, G):
    counterparties = defaultdict(set)
    for u, v in G.edges():
        counterparties[u].add(v)  # u sent to v
        counterparties[v].add(u)  # v received from u

    for i, a in enumerate(addresses):
        for b in addresses[i+1:]:
            shared = counterparties[a] & counterparties[b]
            shared.discard(a); shared.discard(b)
            if len(shared) >= self.min_shared:
                links.append((a, b))

Example: Wallet A and Wallet B both send to Uniswap, Binance deposit, and a known mixer. They share 3 counterparties → linked.

② Smart Contract Interaction contract_interaction Score: +25

Concept: Addresses that interact with the same smart contracts (DeFi protocols, mixers, bridges) are grouped. Additionally, the classic mixer pattern — Deposit → Contract → Withdraw — is detected by finding addresses that deposit into a contract and different addresses that withdraw from it.

Algorithm:

  1. Identify contract nodes: any node that receives a is_contract_call=True edge
  2. Group all callers of each contract together
  3. For each contract, find pure depositors (only send to contract) and pure withdrawers (only receive from contract) — link them as a deposit/withdraw pair
# Detect Deposit → Contract → Withdraw pattern
for contract in contract_nodes:
    depositors = {u for u, v in G.in_edges(contract)}
    withdrawers = {v for u, v in G.out_edges(contract)}
    pure_depositors = depositors - withdrawers
    pure_withdrawers = withdrawers - depositors
    for dep in pure_depositors:
        for wit in pure_withdrawers:
            links.append((dep, wit))  # mixer pattern

Example: Address A deposits 5 ETH into Tornado Cash. Address B withdraws 5 ETH from Tornado Cash. A and B are linked as a possible mixer pair.

③ Token Flow Clustering token_flow Score: +15

Concept: ERC-20 token transfers (identified by method IDs 0xa9059cbb for transfer() and 0x23b872dd for transferFrom()) between the same pair of addresses in both directions, or repeated unidirectional flows, indicate a controlled relationship.

# ERC-20 method IDs
_ERC20_METHODS = {"0xa9059cbb", "0x23b872dd"}

for u, v, data in G.edges(data=True):
    if data.get("input_method_id", "")[:10] in _ERC20_METHODS:
        flow_count[(u, v)] += 1

for (u, v), count in flow_count.items():
    # Bidirectional: A→B and B→A both exist
    if flow_count.get((v, u), 0) > 0:
        links.append(tuple(sorted([u, v])))
    # Repeated unidirectional: same pair, N+ times
    elif count >= _MIN_REPEATED_FLOWS:
        links.append(tuple(sorted([u, v])))

Example: Address A sends USDT to Address B three times. Address B sends USDT back to A once. Bidirectional flow detected → linked.

④ Temporal Coordination temporal_coordination Score: +20

Concept: Multiple addresses sending to the same destination within a very short time window (default: 60 seconds) exhibit coordinated behavior. This is a strong signal of automated scripts, bots, or coordinated attacks.

Algorithm: Sliding window over time-sorted transactions. For each destination, track all senders within the window. If ≥ 3 distinct senders target the same destination within W seconds, link all those senders together.

# Sort all edges by timestamp
edges_ts.sort(key=lambda x: x[0])

# Sliding window
for right in range(n):
    ts_r, u_r, v_r = edges_ts[right]
    receiver_senders[v_r].append((ts_r, u_r))

    # Evict entries outside the W-second window
    while edges_ts[left][0] < ts_r - self.window:
        # remove oldest entry
        left += 1

    senders = {s for _, s in receiver_senders[v_r]}
    if len(senders) >= _MIN_COORDINATED_SENDERS:
        # Link all senders together
        for i, j in combinations(sorted(senders), 2):
            links.append((i, j))

Example: Addresses A, B, C all send ETH to the same exchange wallet within 30 seconds. They are linked as coordinated senders.

⑤ Fan-In / Fan-Out Detection fan_pattern Score: +20

Concept:

  • Fan-out: One address distributes funds to many addresses (≥ threshold, default 5). Common in exchange hot wallets, mixer outputs, and airdrop contracts. All receivers are linked.
  • Fan-in: Many addresses consolidate funds into one. Common in consolidation wallets before mixing. All senders are linked.
# Fan-out: one sender → many receivers
for node in G.nodes():
    out_neighbours = list(set(G.successors(node)))
    if len(out_neighbours) >= self.threshold:
        # Link all receivers together
        for i, j in combinations(sorted(out_neighbours), 2):
            links.append((i, j))

# Fan-in: many senders → one receiver
for node in G.nodes():
    in_neighbours = list(set(G.predecessors(node)))
    if len(in_neighbours) >= self.threshold:
        # Link all senders together
        for i, j in combinations(sorted(in_neighbours), 2):
            links.append((i, j))

Example: A mixer contract sends ETH to 8 different addresses. All 8 receivers are linked as likely belonging to the same entity or campaign.

⑥ Loop / Circular Transaction Detection loop_detection Score: +35 (highest)

Concept: Circular fund flows (A → B → C → A) are the strongest indicator of layering — the second stage of money laundering. They are also used in wash trading to simulate volume. All addresses in a detected cycle are linked.

# Collapse parallel edges for cycle detection
simple = nx.DiGraph(G)

for cycle in nx.simple_cycles(simple):
    if 2 <= len(cycle) <= self.max_depth:  # default max depth: 6
        cycle_sorted = sorted(cycle)
        for i, j in combinations(cycle_sorted, 2):
            links.append((i, j))

Example: A → B → C → D → A forms a 4-hop cycle. All four addresses are linked. Risk score contribution: +35 points.

Performance Note: nx.simple_cycles() can be expensive on large graphs. The max_depth parameter (default 6) limits cycle length to keep runtime manageable.

Union-Find Merge Algorithm

After all heuristics run, the collected address pairs are merged using Union-Find. This ensures that if A-B and B-C are linked by different heuristics, A, B, and C all end up in the same cluster.

class UnionFind:
    def find(self, x):
        # Path compression: flatten the tree
        if self._parent[x] != x:
            self._parent[x] = self.find(self._parent[x])
        return self._parent[x]

    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb: return
        # Union by rank: attach smaller tree under larger
        if self._rank[ra] < self._rank[rb]: ra, rb = rb, ra
        self._parent[rb] = ra
        if self._rank[ra] == self._rank[rb]:
            self._rank[ra] += 1

Time complexity: O(α(n)) per operation where α is the inverse Ackermann function — effectively constant for all practical inputs.

def score_cluster(cluster_id, addresses, heuristics_fired, indicators):

Scalability & UTXO Design

Current Architecture: Account-Based (Ethereum)

In account-based blockchains like Ethereum, each address has a balance and transactions directly debit/credit those balances. This makes clustering straightforward: addresses are nodes, transactions are directed edges.

Future Architecture: UTXO-Based (Bitcoin)

UTXO (Unspent Transaction Output) blockchains like Bitcoin work differently. There are no "accounts" — instead, transactions consume previous outputs and create new ones. The classic UTXO clustering heuristic is common input ownership: all inputs to a transaction are assumed to be controlled by the same entity.

Design Decision: The BlockchainAdapter abstract base class was specifically designed to allow UTXO support without modifying the clustering engine or any heuristics. Only a new adapter needs to be written.

Best Practices & Limitations

Known Limitations

False Positives — Fan-Out Heuristic: Legitimate exchange hot wallets distribute funds to thousands of users. They will trigger the fan-out heuristic and link all recipients together. Mitigation: maintain a whitelist of known exchange addresses and exclude them from clustering.
False Positives — Behavioral Similarity: Popular DeFi protocols (Uniswap, Aave) are counterparties for millions of addresses. Two unrelated users both using Uniswap will share a counterparty. Mitigation: exclude high-degree nodes (degree > threshold) from the counterparty sets.
Loop Detection Performance: nx.simple_cycles() has exponential worst-case complexity on dense graphs. The max_depth parameter (default 6) limits this, but very large graphs may still be slow. Mitigation: run loop detection on subgraphs or use approximate cycle detection.
Pseudonymity ≠ Anonymity: This system identifies behavioral clusters but cannot definitively attribute them to real-world identities without additional off-chain data (exchange KYC, IP logs, etc.). Cluster results should be treated as investigative leads, not proof.

Recommended Improvements

Best Practices for Deployment

  • Run clustering after each ETL batch, not in real-time, to avoid partial-block artifacts
  • Store cluster results with updated_at timestamps to track cluster evolution over time
  • Use --min-size 3 in production to reduce noise from 2-address clusters
  • Monitor the time_span_seconds indicator — clusters active over long periods are more suspicious than short bursts
  • Cross-reference high-risk clusters against OFAC sanctions lists and known mixer addresses
  • Never use clustering results alone as grounds for account freezing — always require human analyst review

Advanced Clustering Heuristics

The following five advanced heuristics extend the clustering engine beyond basic pattern matching into sophisticated AML techniques used by professional blockchain forensics firms.

⑦ Peel Chain Detection peel_chain Score: +40 (highest)

Concept: A peel chain is a sequence of transactions where each address receives funds, forwards most of it to the next address, and keeps a small "peel" (remainder). This creates a long chain of hops that obscures the original source of funds.

A --[10.0 ETH]--> B --[9.8 ETH]--> C --[9.6 ETH]--> D --[9.4 ETH]--> E
                  ↓                 ↓                 ↓                 ↓
               0.2 ETH           0.2 ETH           0.2 ETH           kept

Each node forwards >= 85% of received value → relay node
Chain length >= 3 → peel chain detected
All nodes A,B,C,D,E linked into one cluster
# Identify relay nodes: forward >= 85% of incoming value
for node in G.nodes():
    iv = in_value.get(node, 0.0)
    dominant_val = max(out_edges[node], key=lambda x: x[1])[1]
    if dominant_val / iv >= _FORWARD_RATIO:  # 0.85
        relay_next[node] = dominant_receiver

# Walk the chain and link all members
for start in relay_next:
    chain = walk_chain(start, relay_next)
    if len(chain) >= _MIN_CHAIN_LENGTH:  # 3
        link_all_pairs(chain)

Why it matters: Peel chains are the most common layering technique on Ethereum. They are designed specifically to defeat naive "follow the money" analysis. Detecting them requires looking at the ratio of forwarded value, not just the transaction graph structure.

⑧ Layering & Structuring (Smurfing) layering Score: +38

Concept — Rapid Layering: Funds are received and forwarded within a very short time window (≤5 minutes), creating a rapid multi-hop chain. Unlike peel chains, amounts may change significantly between hops.

Concept — Structuring (Smurfing): A large amount is split into many transactions just below a reporting threshold (e.g., 9.9 ETH × 5 instead of 49.5 ETH × 1) to avoid triggering AML alerts.

Rapid Layering:
  t=0:   A receives 50 ETH
  t=120: A sends 48 ETH to B    (within 5 min → rapid relay)
  t=180: B sends 45 ETH to C    (within 5 min → rapid relay)
  t=240: C sends 43 ETH to D    (within 5 min → rapid relay)
  Chain: A→B→C→D  (length 4 ≥ 3) → all linked

Structuring:
  Sender X sends: 9.9 ETH to A, 9.8 ETH to B, 9.7 ETH to C
  All below 10 ETH threshold, 3+ transactions → A,B,C linked
# Rapid relay: received then forwarded within RAPID_HOP_SECONDS
if send_time[node] - receive_time[node] <= _RAPID_HOP_SECONDS:
    rapid_relay_next[node] = send_to[node]

# Structuring: many sub-threshold transactions from same sender
if 0 < val < _STRUCTURING_THRESHOLD and count >= _STRUCTURING_MIN_COUNT:
    link_all_receivers(sender)

⑨ Community Detection (Louvain-style) community_detection Score: +30

Concept: Community detection finds groups of addresses that transact with each other far more densely than with the rest of the network. This is a graph-level technique that catches clusters no single pair-wise heuristic would find.

Algorithm: Uses NetworkX's greedy modularity communities (approximation of the Louvain algorithm) on an undirected weighted projection of the transaction graph.

Transaction graph (directed, weighted by tx count):

  [A] ←→ [B] ←→ [C]        [X] ←→ [Y]
   ↕      ↕      ↕            ↕
  [D] ←→ [E] ←→ [F]         [Z]

Community 1: {A,B,C,D,E,F}  ← dense internal connections
Community 2: {X,Y,Z}         ← separate dense group

Modularity score measures how much denser these communities are
compared to a random graph with the same degree sequence.
# Project to undirected weighted graph
for u, v, data in G.edges(data=True):
    UG[u][v]["weight"] = UG.get_edge_data(u,v,{}).get("weight",0) + 1

# Run greedy modularity community detection
from networkx.algorithms.community import greedy_modularity_communities
communities = greedy_modularity_communities(UG, weight="weight")

for community in communities:
    if len(community) >= _MIN_COMMUNITY_SIZE:
        link_all_pairs(community)

Why it matters: Wash trading rings and coordinated wallet clusters often have no single obvious pattern — they just transact heavily with each other. Community detection catches these without needing a specific behavioral signature.

⑩ Address Poisoning Attack Detection address_poisoning Score: +28

Concept: An attacker creates an address that looks visually identical to a victim's frequent counterparty (same first 6 and last 4 characters), then sends a zero-value transaction to the victim. The victim, copying from their transaction history, accidentally sends funds to the attacker's address.

Legitimate address:  0xABCDEF...1234  (victim's real counterparty)
Poisoning address:   0xABCDEF...1234  (attacker's lookalike)
                     ^^^^^^    ^^^^
                     same      same  ← visually identical in most UIs

Attack flow:
  1. Attacker sends 0 ETH from poisoning address to victim
  2. Victim sees "0xABCDEF...1234" in their history
  3. Victim copies address and sends real funds to attacker
# Build (prefix, suffix) lookup for all addresses
for addr in nodes:
    clean = addr.lower().replace("0x", "")
    key = (clean[:6], clean[-4:])  # first 6 + last 4 chars
    prefix_suffix_map[key].append(addr)

# Flag pairs with same prefix+suffix AND zero-value transaction
for similar_addrs in prefix_suffix_map.values():
    if len(similar_addrs) >= 2 and has_zero_value_edge(similar_addrs):
        link_all_pairs(similar_addrs)

⑪ Dusting Attack Detection dusting_attack Score: +25

Concept: A dusting attack sends tiny amounts of cryptocurrency (< 0.001 ETH) to many addresses. The attacker then monitors when those addresses spend their funds. By tracing the dust, they can de-anonymize wallet owners and link addresses to real identities.

Dust sender (attacker):
  → 0.0001 ETH to Wallet_A
  → 0.0001 ETH to Wallet_B
  → 0.0001 ETH to Wallet_C
  → 0.0001 ETH to Wallet_D
  → 0.0001 ETH to Wallet_E  (5+ recipients → dusting attack)

All recipients A,B,C,D,E are linked as targets of the same
surveillance campaign. The attacker is also flagged.
# Find dust senders: many tiny-value transactions
for u, v, data in G.edges(data=True):
    if 0 < val <= _DUST_THRESHOLD_ETH:  # 0.001 ETH

Why it matters: Dusting attacks are used by both surveillance firms and malicious actors. Detecting them helps identify which addresses are being actively tracked and may indicate that those addresses are of interest to threat actors.
